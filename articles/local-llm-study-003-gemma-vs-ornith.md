## 前回のおさらい

前回は、Aider + Ollamaで7モデルを比較し、単純なバグ修正タスクでは `gemma4:e2b`（gemma4の軽量版、7.2GB）と `Ornith-1.0-9B`（コーディング特化・MITライセンス、6.9GB）が特に有望という結論になりました。もう1つ、`qwen3:14b`（9.3GB）も改竄・誤修正は一度も起きず信頼性は問題なしでしたが、実行速度が他モデルの3〜5倍遅いという理由だけで、その時点の最終候補には含めていませんでした。

「速度以外に弱点がない」という評価が、より複雑なタスクでも通用するのかを確かめるため、今回は `qwen3:14b` を含めた3モデルで、実際の開発フローに近い「計画→実装+テスト→ドキュメント化」という3段階のエンドツーエンドタスクを実施しました。単純な1関数バグとは違い、モデル自身にテストを一から設計させる場面を作るのが狙いです。

## 検証方法：3段階プロセス

題材はカルマンフィルタ（センサーノイズから位置・速度を推定するアルゴリズム）と、特異値分解（べき乗法による実装）の2つです。`gemma4:e2b` と `Ornith-1.0-9B` は次の3段階で進めました。

1. **計画**：コードは書かず、`PLAN.md` に設計方針だけを書かせる
2. **実装+テスト**：計画に基づき実装ファイルとテストファイルを、`--auto-test` で自律的に完成させる
3. **ドキュメント化**：実装内容をもとにHTMLスライドを作成させる

`qwen3:14b` も同じ3段階・同じ指示文で検証しましたが、1点だけ手法が異なります。study002の時点ですでに実行速度の遅さが分かっていたため、`--auto-test` による自動リトライループは使わず、各段階を1回の生成で完了させ、その結果を人間が直接検証しました。ループによる自動修正の機会がない分、モデルの「一発品質」がそのまま結果に出ています。この違いは後述の結論にも関わる、重要な注記です。

なお、ローカルモデルにインターネット検索機能はないため、「調べて実装して」という依頼は実質的に各モデルの内部知識に基づく設計を意味します。

## カルマンフィルタ：実装は3モデルとも正確、明暗を分けたテスト作成

kalman.py本体の実装は3モデルとも1回で正確でした。数式・行列演算に誤りはありません。差が出たのは**テストファイルの自作**でした。

`gemma4:e2b` に「`unittest.TestCase` を使い、`self.assertXxx()` で実際に検証すること」と明示して依頼したところ、1〜2回目はテストではなく `print` 文だけのシミュレーションスクリプトを書きました。

```python
# gemma4:e2bが最初に書いたもの（test_kalman.pyのつもり）
def run_simulation():
    ...
    print(f"Mean Squared Error (MSE): {mse:.4f}")

if __name__ == '__main__':
    run_simulation()
```

`unittest.TestCase` の骨格を明示的に与えても、3〜5回目はメソッドの中身が `pass` のまま変化しませんでした。

```python
class TestKalmanFilter1D(unittest.TestCase):
    def test_converges_to_true_position(self):
        pass  # 何度指示しても中身が埋まらない
```

厄介なのは、`--auto-test` が「0件のテストが成功」または「空のテストが成功」を**正当な合格と誤認してループを終える**ことです。見た目には「OK」と表示されるため、気づかずに進めてしまうリスクがあります。

**6回目、チャット履歴をリセットした新規セッションで、全く同じ指示を出したところ1回で解決しました。** 同一セッション内に蓄積した失敗のやり取りが、モデルを誤った応答パターンに固着させていたようです。これは実用上重要な知見で、Aiderで2〜3回訂正しても改善しない場合は、粘らずセッションを切り直す方が効率的だと分かりました。

一方 `Ornith-1.0-9B` は386秒・2回のループで、`unittest.TestCase` ベースの4テスト（収束・共分散減少・predict単体・制御入力付きpredict）を一発で生成しました。すべて実質的な検証を含み、空虚な成功は皆無でした。しかもテストの厳密さ自体もgemma4:e2b（最終的に人間が書いたコード）より上でした——収束テストで位置だけでなく速度も検証し、共分散も対角成分ではなく `np.trace(P)` で全体的な不確実性を見ています。

Ornithの唯一のミスは、指定した `slides.html` ではなく `kalman_slides.html` という別名で保存してしまったことです。手動でリネームして対応しました。

### qwen3:14b：実装は一発で正確、しかし自作テストに2つの見落とし

qwen3:14bは計画（PLAN.md）を168秒で書き上げ、状態遷移・観測モデルの数式やクラス設計は過不足ありませんでした。続く実装+テストの生成には**5820秒（97分）**かかり、`kalman.py` の実装自体は数式的に完全に正しいことを検算で確認しました。

ただし自作した `test_kalman.py` は3件中2件が失敗していました。

```
FAIL: test_covariance_decreases
FAIL: test_converges_to_true_position
```

原因を1つずつ検算しました。

- `test_covariance_decreases`：`update()` を1回呼んだだけで位置・速度両方の分散が減ることを期待していましたが、観測モデルが `H=[1, 0]`（位置のみ観測）である以上、対角共分散（位置と速度が無相関）から出発した状態で速度側の分散が1回のupdateで減ることは数学的にあり得ません。predict+updateを10サイクル回すよう修正したところ解決しました。
- `test_converges_to_true_position`：観測ループ内で `update()` だけを呼び `predict()` を呼び忘れており、推定位置が真値から大きく遅れる実装（真値110.0に対し推定60.7）になっていました。ループ内に `predict()` を追加したところ解決しました。

どちらも**kalman.pyの実装ではなく、qwen3:14b自身が書いたテストの前提に誤りがあった**という点で、後述する「存在しないバグの追跡」と同じパターンです。手動で2箇所を修正した後は3/3 PASSしました。

スライド生成（Stage3）は397秒で完了し、日本語・ページカウンタ・キー操作いずれも問題ありませんでした。

qwen3:14bの弱点は「テストが書けない」ことではなく、**自作テストの数学的前提を検算せずに確信を持って書いてしまう**ことのようです。これはgemma4:e2bのSVDタスクで見られたパターンと本質的に同じで、後段で改めて整理します。

## SVD（ランク1近似）：3モデルを比較

次に、べき乗法とデフレーションによるランクk特異値分解を依頼しました。ところがこちらは `gemma4:e2b`・`Ornith-1.0-9B` ともに**深刻な失敗**に陥りました。

- **Ornith-1.0-9B**：計画段階で2回連続、10分待っても応答なし。3回目でようやく応答したものの、「計画だけ書いて、コードは書かないで」という指示を無視し、実装・テスト・スライドまで一気に生成しようとしました。しかも `--edit-format whole` のはずが unified diff形式（`@@ ... @@`）で出力したため、Aiderがパースできずファイルはすべて0バイトのまま終わりました。
- **gemma4:e2b**：実装段階で20分以上生成を続けても完了しませんでした。

デフレーション（2番目以降の特異値を求める反復処理）を除き、「最大特異値・特異ベクトルのみをべき乗法で求める（ランク1近似）」に縮小して再試行したところ、両モデルとも問題なく完走しました。SVDというテーマ自体ではなく、複数階数を反復的に求める設計の複雑さが両モデルの限界を超えていたようです。

`qwen3:14b` については、study002ですでに判明していた低速傾向を踏まえ、最初からこのランク1近似版のみを検証しました（フルスペック版は未検証です）。

### gemma4:e2bが「存在しないバグ」を4回追いかけた話

簡素化後、gemma4:e2bの実装は536秒・4回のループを費やしても、あるテストだけ失敗し続けました。

```
AssertionError: 0.8902277713535564 not less than 1e-05
```

実装コードを見直しても間違いが見当たらなかったため、テストで使っている行列 `A = [[10, 1], [1, 1]]` の特異値を実際に計算してみました。

```python
>>> import numpy as np
>>> A = np.array([[10.0, 1.0], [1.0, 1.0]])
>>> np.linalg.svd(A)[1]
array([10.10977223, 0.89022777])
```

テストのコメントは「特異値は約10と0（ほぼランク1）」となっていましたが、実際には**第2特異値が0.89もある**、まともにランク1近似できない行列でした。実測誤差 `0.8902277713535564` は、この行列のランク1近似誤差の理論的下限（Eckart-Young定理、第2特異値そのもの）とほぼ完全に一致します。

つまり **gemma4:e2bのべき乗法の実装は最初から正しく、直しようのないテストを4回にわたって追いかけ続けていた**ことになります。テストの行列を真にランク1のもの（2ベクトルの外積）に差し替えたところ、即座に4/4 PASSしました。

これは前述の「空虚な成功」とも、以前見つけた「テスト改竄」とも違う、3つ目の新しい失敗パターンです。`--auto-test` は「実装が正しいか」しか見ておらず、「テストの前提が正しいか」は誰もチェックしていない、という盲点を突いた形です。

### qwen3:14b：正しい実装を、テストの許容誤差が裏切った話

qwen3:14bは計画を495秒、実装+テストを1518秒（25分）で完了しました。`power_iteration_svd` の実装内容を確認したところ、標準的なべき乗法（$A^TAv$ を正規化しながら反復し、収束判定は `||new_v - v|| < 1e-6`）で、ロジックに誤りはありません。

ところが自作テストが4件中1件、確率的に失敗しました。

```
AssertionError: 1.0117398783180041e-06 != 0.0 within 1e-06 delta
```

5回実行して都度誤差の大きさを記録したところ、2.08e-07〜1.01e-06の範囲でばらついており、**テストの許容誤差（delta=1e-6）が、実装自身の収束判定基準（1e-6）とほぼ同じオーダー**だと分かりました。反復を1e-6の変化量で打ち切る以上、真値との残差がその前後になるのは当然で、テストのdeltaがピッタリ同じ値である限り、確率的に成功したり失敗したりするテストになっていたのです。

deltaを1e-4に緩めたところ、5回連続で安定してPASSしました。gemma4:e2bのSVDタスクで見つかった「存在しないバグの追跡」と同種ですが、1回の実行だけでは「たまたま通った」のか「実装が正しい」のか判別できない**確率的な失敗**という点で、より発見しづらいバリエーションです。

スライド生成（Stage3）は3026秒（約50分）かかりましたが、内容・フォーマットともに問題ありませんでした。

## 4つの失敗パターン一覧

ここまでの検証を通じて、7モデル・14タスク以上で見つかった失敗は、すべて「実装ロジックの誤り」ではなく、その周辺で起きていました。

| パターン | モデル | 内容 | 気づきやすさ |
|---|---|---|---|
| ①テスト改竄 | qwen2.5-coder:7b | バグは直さず、テストの境界値をずらして偽装合格 | 静か（diffを見ないと分からない） |
| ②空虚な成功 | gemma4:e2b | テスト本体が`pass`のまま、または検証なしのシミュレーションスクリプト | 静か（テスト内容を読まないと分からない） |
| ③存在しないバグの追跡 | gemma4:e2b、qwen3:14b（2回） | テストの前提が誤っており、正しい実装を延々と疑い続けた | 静か（テストの前提を検算しないと分からない） |
| ④スコープ逸脱・フォーマット崩壊 | Ornith-1.0-9B | 指示を無視して先の段階まで生成し、出力フォーマットも崩れてファイルが空に | 派手（ファイルが空／変な名前で一目瞭然） |

qwen3:14bはカルマンフィルタとSVDの両方で③のパターンを踏んでおり、検証した7モデルの中でもっとも③に該当した回数が多いモデルになりました。gemma4:e2bとqwen3:14bの失敗（②③）は「動いているように見えて中身がない、あるいは前提が壊れている」という共通点があり、CI的なチェック（テストがPASSしたか）だけでは見逃します。Ornithの失敗（④）はファイルが空になる・名前が違うという形で即座に露見します。自動化パイプラインの危険度は発生頻度だけでなく「気づきやすさ」でも測るべきで、この観点では静かな失敗を出すモデルの方が慎重な扱いが必要になります。

## 結論：Ornith-1.0-9Bを既定モデルにする

| 比較軸 | Ornith-1.0-9B | gemma4:e2b | qwen3:14b |
|---|---|---|---|
| テストを自分で設計する力 | 一貫して高品質・高速 | 2フェーズとも大きくつまずいた | 実装は毎回正確だが、自作テストが2回とも静かに誤っていた |
| 生成速度 | 安定して速い（386秒程度） | 安定して速い | 極端かつ不規則に遅い（168秒〜5820秒、自身の実行間で最大約35倍のばらつき） |
| 失敗時の気づきやすさ | 派手に壊れる＝気づきやすい | 静かに壊れる＝見逃しやすい | 静かに壊れる＝見逃しやすい |
| モデルサイズ | 6.9GB | 7.2GB（ほぼ同等） | 9.3GB |
| 検証手法 | `--auto-test`の自動リトライループあり | `--auto-test`の自動リトライループあり | ループなし・単発生成を人間が検証 |

特に「テストを一から設計させる」タスクではOrnithが明確に優位でした。gemma4:e2bはこの種のタスクで2回つまずき、qwen3:14bも2回とも静かなテストの誤りを起こしています。

qwen3:14bはstudy002の時点で「信頼性は問題なし、除外理由は速度のみ」としていましたが、**今回の複雑なタスクではその前提が崩れました**。実装自体は3モデル中で最も一貫して正確だったものの、自作テストの数学的前提を検算しない傾向はgemma4:e2bと同じ「静かな失敗」のリスクを抱えています。加えて生成速度も、単純タスクで見えていた「3〜5倍遅い」という水準を大きく超え、1回の生成に97分かかるなど予測不能でした。なお、qwen3:14bのみ`--auto-test`の自動リトライループを使わず単発生成で検証しているため、ループを使えば自己修正できていた可能性がある点は公平のため付記します。

Ornithの唯一の重大リスク（スコープ逸脱・フォーマット崩壊）は、**タスクを能力範囲内に留める**ことで大きく抑制できます。SVDタスクの簡素化前後の結果がそれを裏付けています。段階を分けて依頼する、複雑なタスクは事前に簡素化する、という運用で発生率を下げられそうです。

とはいえ、Ornithの失敗が派手で気づきやすいとはいえ、無人パイプラインでは「気づく人間がいない」ことが前提になります。以下のガードを組み合わせることで、4パターンすべてに対する防御線になります。

| ガード | 防ぐ失敗パターン |
|---|---|
| テスト改竄検知（実行前後のdiffチェック） | ①テスト改竄 |
| テストファイルのカバレッジ確認（0件・空メソッドの検出） | ②空虚な成功 |
| テストの前提を独立に検算する | ③存在しないバグの追跡 |
| 大きなタスクは段階分割し、各段階の出力ファイルを確認する | ④スコープ逸脱・フォーマット崩壊 |

## この結論が変わりうる条件

この推奨は7モデル・14タスク以上という限られたサンプルに基づいています。特に次の点は今後の追加検証で変わりうると考えています。

- Ornith-1.0-9BのSWE-bench Verified 69.4は自社発表で、第三者検証は未了
- 今回の複雑タスクはすべて数値計算・アルゴリズム系。Web API連携やUIコードでは順位が変わる可能性がある
- gemma4:e2bの「テスト設計が苦手」という傾向は2フェーズのみの観測。単純作業への強さ（前回記事で最速だった実績）は評価に値し、用途を絞れば依然有力
- qwen3:14bは`--auto-test`の自動リトライループを使わず単発生成で検証したため、他モデルと完全に同一の手法ではありません。ループを使えば自己修正できていた可能性があります
- 16GB統合メモリという制約下での比較であり、大きなモデルほど不利なバイアスがかかっている可能性がある

新しいモデルが出た際は、今回と同じ3段階プロセスで追試していく予定です。
