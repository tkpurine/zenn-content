## はじめに

クラウド型のAIコーディングアシスタントは便利ですが、月額コストやコードの外部送信が気になる場面があります。ちょうどGitHub Copilotの値上げ通知が届いたこともあり、「話題のローカルLLMは本当にコーディング用途で使えるのか」を確かめてみたいと思い、MacBook Airの上で動くモデルを片っ端から試すことにしました。

この記事では、Ollamaで動かせる11モデルを同じお題で比較した実測結果をまとめます。単発のQ&A（コードレビューやFizzBuzz生成、バグ検出、日本語説明）で、どこまで実用に耐えるかを見ています。

本記事（study1）はローカルLLM全体の見取り図として広く11モデルを比較します。次回（study2）は有望なモデルに絞って [Aider](https://aider.chat/) による自律的なバグ修正・テスト作成まで検証します。新しいモデルが出れば、このシリーズは今後も追加していく予定です。

> **結論だけ先に知りたい方へ：** Copilot代替として普段使いするなら `gemma4:e2b`（速度・品質・日本語のバランスが最良）、説明やレビューの質を優先するなら `gemma4:e4b` か `Ornith-1.0-9B` がおすすめです。詳しくは後半の「用途別おすすめ」の章へ。

## 検証環境

| 項目 | 内容 |
|---|---|
| 端末 | MacBook Air（Apple **M4**） |
| メモリ | 16GB |
| macOS | 26.5.1 |
| Ollama | v0.30.6 |
| 検証時期 | 2026年6月 |

**この記事の数値をそのまま自分の環境に当てはめる前に：** チップ世代（M1〜M4）やメモリ容量が違うと、同じモデルでも速度が大きく変わります。特にMacBook Airはファンレス構造のため、連続してモデルを走らせると発熱でパフォーマンスが落ちることがあります（後述の「負荷再検証」も参照）。自分のMacのチップとメモリは、ターミナルで次のコマンドで確認できます。

```sh
# チップの種類
sysctl -n machdep.cpu.brand_string

# メモリ容量
sysctl -n hw.memsize | awk '{print $1/1024/1024/1024 " GB"}'
```

**計測条件についての正直な注記：** 今回の検証中盤で「Ornith-1.0-9Bだけ他の作業と並行して計測してしまい、速度が大きく歪んだ」ことが判明し、Ornithのみ負荷のない状態で再計測しました（詳細は速度まとめの章）。他の10モデルについては、この教訓を踏まえた上での再検証は行っておらず、多少のバックグラウンド負荷がかかっていた可能性があります。絶対値よりも「モデル間の相対的な傾向」として読んでいただくのが安全です。

**Intel Macでも使えるか：** Ollama自体はIntel Macでも動作します。ただし、Apple SiliconのGPUコア（Metalによる高速化）を活用できないため、CPU中心の実行になり大幅に遅くなります。一部のIntel Mac（MacBook Pro 15/16インチの一部モデルなど）はディスクリートGPUを搭載していますが、VRAMが4〜8GB程度と小さく、今回の7B〜14Bクラスのモデルは載り切らないことが多いです。Intel Macをお持ちの場合は、まず軽量な `qwen2.5-coder:7b` あたりから試すことをおすすめします。

### 用語ミニ解説

- **`7b` `14b` などの数字（B = Billion）**：モデルのパラメータ数です。大きいほど基本的に賢くなりますが、その分メモリを食い、速度も遅くなります。`7b`は「70億パラメータ」という意味です。
- **Thinking（思考）モデル**：回答を出す前に、内部で「考える過程」を出力してから最終回答を返すモデルです。精度が上がることもありますが、その分時間がかかります。
- **量子化**：本来もっと重いモデルを、精度を多少犠牲にして軽量化する技術です。Ollamaで配布されているモデルの大半はこの量子化済みバージョンで、「7.2GB」のようなサイズ表記は量子化後の値です。

`gemma4` の `e2b` `e4b` はパラメーター数の略ではありません。実際は5.1Bと8.0Bで、画像・音声・ツール呼び出しにも対応したマルチモーダルモデルです。

## Ollamaのセットアップ

インストールから実行まで、拍子抜けするほど簡単でした。

```sh
# インストール
brew install ollama

# サービス起動
ollama serve

# モデルをダウンロードして即チャット開始
ollama run gemma4:e2b
```

APIキー不要、サインアップ不要で `ollama run モデル名` を叩くだけです。ダウンロードサイズの目安は、qwen2.5-coder:7bが4.7GB（数分）、gemma4:e2bが7.2GB（5〜10分）、qwen2.5-coder:14b・gemma4:e4bが9〜9.6GB（10〜15分）でした。

### つまずきやすいポイント

実際に触ってみて、最初に戸惑いそうな点をまとめておきます。

| 症状 | 原因・対処 |
|---|---|
| `ollama: command not found` | Homebrewのパスが通っていないことが多いです。ターミナルを再起動するか、シェルの設定ファイルを読み込み直してください |
| `ollama serve` で `bind: address already in use` エラー | [Ollama公式サイト](https://ollama.com/)からGUIアプリ版をインストール済みだと、すでにバックグラウンドでサーバーが起動しています。この場合 `ollama serve` を手動実行する必要はなく、そのまま `ollama run モデル名` で使えます |
| `ollama run` 実行後、数分間何も表示されない | フリーズではなく、数GB規模のモデルダウンロードが進行中です（進捗バーが出ないことがあります）。回線速度によっては本当に数分かかります |
| モデル名を打ち間違えて別モデルが動く | `gemma4:e2b` のつもりで `gemma:2b`（無関係の別モデル）のように似た名前を打ち間違えやすいです。コピペを推奨します。今何が入っているかは `ollama list` で確認できます |
| 動作が異常に遅い・固まる | メモリ不足でスワップが発生している可能性があります。ブラウザのタブを大量に開いていたり、他の重いアプリを同時に動かしていないか確認してください（前述の「バックグラウンド負荷」の話と直結します） |

**GPUが使われているか確認する方法：** モデル実行中に別のターミナルで `ollama ps` を打つと、`PROCESSOR` 列に `100% GPU` のように表示されます。ここが `CPU` になっている場合は、GPUアクセラレーションが効いておらず、本来より大幅に遅くなっている可能性があります。

**ディスク容量が気になったら：** 今回のように複数モデルを試すと、あっという間に数十GB消費します。使わなくなったモデルは `ollama rm モデル名` で削除できます。`ollama list` で一覧を見てから整理するのがおすすめです。

## タスク1：コードレビュー

```python
def divide(a, b):
    return a / b
```

このシンプルな関数をレビューさせ、どこまで気づけるかを見ました。ゼロ除算バグの指摘は全モデルが正解でしたが、そこから先で差が出ました。

| 提案内容 | 該当モデル |
|---|---|
| 型チェック（isinstance）の提案 | qwen2.5-coder:14b・gemma4シリーズ・Ornith-1.0-9B |
| 型アノテーションの指摘 | gemma4シリーズ・Claude Code |
| 引数名の改善提案（dividend/divisor） | Claude Codeのみ |
| 深刻度の分類（表形式） | Claude Code・Ornith-1.0-9B |
| 段階別改善案（3段階以上） | gemma4シリーズ・Ornith-1.0-9B |

`Ornith-1.0-9B` は「基本ガード」「型チェック付き」「Python 3.10+のmatch文を使った実装」という3バージョンを提示した上で、機能性・エラーハンドリング・ドキュメント・テスト可能性を❌⚠️で採点する表までつけてきました。Claude Codeの「深刻度分類」に最も近い構造化度合いです。

Claude Codeは「深刻度を表で分類する」という視点が独特で、最もエンジニアリング的な印象でした。

## タスク3：バグ検出（文脈理解テスト）

```python
def calculate_average(numbers):
    total = 0
    for n in numbers:
        total += n
    return total / len(numbers)   # 空リスト時にZeroDivisionError

results = []
for i in range(5):
    results.append(i * 2)         # このコードでは実際には発動しない
print(calculate_average(results))
```

このタスクのポイントは「バグはあるが、このコードでは発動しない」という点です。潜在バグに気づくだけでなく、「今の呼び出しコードでは問題ない」という文脈まで読み取れるかを見ました。

文脈理解まで成功したのは7モデルです。

- qwen2.5-coder:14b — 33秒。「resultsが5要素あるので今回は発動しない」と明示
- gemma4:e2b — 35.6秒。同様に文脈を正確に把握
- gemma4:e4b — 58.8秒。`sum()` を使ったPythonicな書き換えも提案
- Ornith-1.0-9B — 69秒。「合計は20で平均は4になる」と実際に計算した上で、「バグはないが、空リストへの防御がない点に注意」と正確に整理
- qwen3.5:9b — 71.2秒
- gemma4:12b — 93秒
- Claude Code — `raise ValueError` という最も意味的に正確な修正案

一方、バグを見逃したモデルもありました。`granite3.3:8b` はZeroDivisionErrorを指摘せず、`codegemma:7b` は「totalが計算されていない」という誤指摘をした上に修正コードが元と一字一句同じというハルシネーションが出ました。`starcoder2:7b` は全く無関係な出力でした。速いモデルほど分析系タスクでは信用しすぎない方がよさそうです。

## モデル別総評

**qwen2.5-coder:7b（★★★☆☆）**：平均16秒。軽くて速く、コード生成は正確です。バグ検出の文脈理解は苦手で、修正案が意味的におかしくなることがありました。まずローカルLLMを試したい人向けの入り口として最適です。

**qwen2.5-coder:14b（★★★★☆）**：平均40秒。バグ検出で文脈まで正確に読み取り、型チェックの提案もできました。9GBとMacBook Airで動かせるギリギリのサイズ感ですが、実際には快適に動きました。

**qwen3.5:9b（★★★☆☆）**：平均91秒。Thinkingモードで推論過程を見せてくれますが、Thinkingでは正しい判断をしたにもかかわらず、コードには存在しない型名を書いてしまうケースがありました。Thinkingの質と出力の質が乖離することがある点は注意が必要です。

**granite3.3:8b（★★☆☆☆）**：FizzBuzzの出力形式が独特で、バグ検出ではゼロ除算を見逃しました。コーディング特化ではないため、今回の用途には合いませんでした。

**codegemma:7b（★★☆☆☆）**：平均12.8秒で全体最速。コード生成だけなら正確ですが、バグ検出でハルシネーションが出て信頼できませんでした。

**starcoder2:7b（★☆☆☆☆）**：チャット形式には設計上対応しておらず、内部トークンが露出したり無関係なコードを大量に出力します。補完専用途向けです。

**gemma4:e2b（★★★★☆・平均34.8秒・必要メモリ約6.7GB）**：実パラメーターは5.1B。コードは動作するものを出し、文脈理解も正確です。日本語説明でも自然な表現を使います。qwen2.5-coder:14bより速く、品質は同等以上でした。

**gemma4:e4b（★★★★★・平均55.2秒）**：実パラメーターは8.0B。デコレーターの説明が全テスト中最も詳細で、Thinkingの内容がそのまま高品質な出力になる一貫性を感じました。

**gemma4:12b（★★★★☆・平均93秒）**：品質は高いですが、e4bとの差が小さく速度は大きく劣ります。

**Ornith-1.0-9B（★★★★★・平均78秒・必要メモリ約6.9GB）**：内容の質は今回のテストで最上位クラスです。コードレビューでは3段階の改善案と評価表、デコレーター説明では複数の実例とまとめ表と、いずれもClaude Code並みに構造化された回答を返しました。負荷のない状態での実測では速度もqwen3.5:9bやgemma4:12bより速く、gemma4:e4b（55.2秒）に次ぐ位置です。バックグラウンド負荷があると大きく遅くなるため、他の作業と並行して使う場合は注意してください。

**Claude Code（sonnet-4-6、★★★★★）**：型アノテーション・命名・深刻度分類まで踏み込む、最もエンジニアリング的な視点でした。ただしコードは外部サーバに送信されます。

## 用途別おすすめ

| 用途 | おすすめ | 理由 |
|---|---|---|
| とにかく速くコードを生成したい | codegemma:7b | 平均13秒、説明不要なら十分 |

| 品質最優先でプライバシー不要 | Claude Code | 圧倒的な構造化と命名センス |

## ローカルで動かす最大のメリット：プライバシー

速度・品質だけでなく、ローカルLLMを選ぶ大きな理由がプライバシーです。GitHub CopilotやClaude Codeはコードを外部サーバに送信しますが、Ollamaはすべてローカルで完結します。

会社の規定でコードの外部送信が禁止されている、`.env` ファイルやAPIキーが近くにある状態でコードを相談したい、NDAがあるクライアントのコードを扱っている——こういう場面ではローカルが安心です。他のクラウドAPIに切り替えても外部サーバに送ることに変わりはないので注意が必要です。

## 必要メモリの目安（Apple Silicon Mac）

`gemma4:e2b` の実測値は6.7GB。16GB Macならこの後の余裕も十分でした。

## まとめ

- ローカルLLMは「使える」レベルに達しています。`gemma4:e2b` と `qwen2.5-coder:14b` はバグ検出の文脈理解まで正確にこなしました
- Copilot代替として最もおすすめは `gemma4:e2b`。平均35秒・必要メモリ6.7GB・動くコードを出す・日本語も自然です
- 説明・レビューを重視するなら `gemma4:e4b` か `Ornith-1.0-9B`。ともに55〜78秒程度で、説明の質はトップクラスでした
- ローカルLLMのベンチマークは**バックグラウンド負荷の影響を強く受けます**。Ornith-1.0-9Bは他の作業と並行させた初回計測で28分もかかったタスクが、負荷のない状態では221秒まで縮みました
- 「コーディングモデル」でも補完専用とチャット対応は別物です。`starcoder2:7b` でハマりました
- プライバシーが必要な用途ではローカル一択です

## 今すぐ試すなら：最初の一歩

読んだだけで終わらせず、実際に触ってみたい方は次の3ステップだけで始められます。
