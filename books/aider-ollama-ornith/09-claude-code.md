---
title: "Claude Codeと組み合わせる"
---

ここまではOrnith-1.0-9BをAider単体で使う方法を扱ってきました。本章では、Claude Codeのようなクラウド型AIコーディングツールを普段使いしている前提で、**その中の一部の作業だけをOrnith-1.0-9Bにオフロードする**という運用について扱います。

## なぜ単体で使わず組み合わせるのか

Claude Codeは強力ですが、定型的な作業（単純なバグ修正、境界値の見直し、一括リネームの検証など）まで毎回クラウドのトークンを使うのはコスト面でもったいないことがあります。かといって、すべてをローカルLLMに任せるのは、Ornith-1.0-9Bにも8章で見た「スコープ逸脱」「複雑タスクでの失速」といった弱点がある以上、リスクが残ります。

現実的な落としどころは、**Claude Codeが全体の設計・判断を担い、切り出しやすい定型作業だけをOrnith-1.0-9Bに委任し、結果をClaude Codeが検証する**という分業です。

## 委任に向くタスク・向かないタスクの基準

| 委任に向く | 委任に向かない |
|---|---|
| 対象ファイルが1〜2個に閉じている | 複数ファイル・複数モジュールにまたがる設計変更 |
| 成功条件が既存テストで明確に判定できる | 「正しさ」の基準があいまい・要件がまだ固まっていない |
| バグの内容が局所的（1関数・1クラス程度） | アルゴリズム全体の再設計が必要 |
| 失敗しても被害が小さい（本番に直接影響しない） | セキュリティ・認証まわりなど失敗の影響が大きい |
| 「計画→実装→確認」のように段階分割できる | 段階分割が難しい、一気通貫でしか成立しないタスク |

8章のベンチマークでも、単純なバグ修正（6モデル比較で62秒・1回で完全解決）では高い信頼性を示した一方、複雑なSVDタスクでは指示を無視して先の工程まで生成しようとするなど不安定になりました。**タスクの大きさを見誤らないことが、委任運用における最大の勘所**です。

## 安全装置：テスト改竄検知ラッパー

Ornith-1.0-9B自体に改竄の実績はありませんが（8章参照）、無人・半自動で運用する以上、モデルを問わず機械的な安全装置を挟んでおくのが安全です。`--auto-test` の実行前後でテストファイルの差分を取り、意図しない変更があれば異常終了させるラッパースクリプトです。

```sh
#!/usr/bin/env bash
# aider-safe-fix.sh: --auto-testの前後でテストファイルの変更を検知する

TEST_CMD="$1"; MESSAGE="$2"; shift 2
FILES=("$@")

# テストファイルを実行前にスナップショット
SNAP_DIR=$(mktemp -d)
for f in "${FILES[@]}"; do
  [[ "$(basename "$f")" == test_* ]] && cp "$f" "$SNAP_DIR/$(basename "$f").before"
done

aider --model ollama/ornith-1.0-9b-ctx32k --edit-format whole \
  --no-auto-commits --yes --no-stream \
  --test-cmd "$TEST_CMD" --auto-test \
  "${FILES[@]}" --message "$MESSAGE"

# 実行後に差分チェック
for f in "${FILES[@]}"; do
  base=$(basename "$f")
  [[ "$base" == test_* ]] || continue
  diff -q "$SNAP_DIR/$base.before" "$f" > /dev/null 2>&1 || {
    echo "🚨 テストファイルが変更されました: $f"
    exit 1
  }
done
echo "✅ テストファイル無改変。修正は正当と判断されます。"
```

このスクリプトは実際にテスト改竄の実データに対して走らせ、正しく検知できることを確認済みです（詳細はstudy002）。モデル名を差し替えるだけで、他のローカルモデルにもそのまま流用できます。

## Claude Codeからの呼び出しワークフロー

Claude CodeにはBashツールで任意のシェルコマンドを実行させられるため、上記のラッパーをそのまま呼び出す形で委任できます。実際の流れはこうです。

1. Claude Codeが、委任に向くタスクだと判断する（前述の基準に照らして）
2. Claude Codeが `aider-safe-fix.sh` を、テストコマンドと具体的な指示文つきで実行する
3. スクリプトの終了コード・出力（PASS/FAIL、改竄検知の有無）をClaude Codeが読み取る
4. **「PASSした」という報告だけで終わらせず、Claude Code自身が `git diff` で実装コードの中身も確認する**（8章で確認した「派手に壊れる」失敗パターンをここで拾う）
5. 問題なければ人間に差分を提示してコミットを促す。問題があれば、Claude Codeが自分で修正するか、指示を変えて再委任する

```sh
./aider-safe-fix.sh \
  "python3 -m unittest test_calc -v" \
  "テストが失敗しています。原因を特定して修正してください。" \
  calc.py test_calc.py
```

ポイントは、**Claude Code自身がOrnith-1.0-9Bの出力を無条件に信用しないこと**です。「テスト改竄がないか」「実装ロジックが本当に正しいか」「指示範囲を超えて余計なファイルまで生成していないか」の最終チェックは、常にClaude Code（あるいは人間）側の役割として残しておきます。

## 委任時の指示文（`--message`）のコツ

Claude Codeからタスクを切り出して渡す際、指示文が曖昧だとOrnith-1.0-9Bのスコープ逸脱を誘発しやすくなります。

- **対象範囲を明示する**：「このファイルだけを直してください」「他のファイルは変更しないでください」と明示的に書く
- **完了条件を明示する**：「テストが全件PASSしたら完了です。それ以上の変更は不要です」と書き、暗黙のゴールポストの先読みを防ぐ
- **段階を1つずつに区切る**：「計画→実装→ドキュメント化」を1回の指示にまとめず、7章のように段階ごとに別コマンドで依頼する

```sh
# 良い例：範囲と完了条件が明確
--message "calc.pyのin_range関数のバグを直してください。test_calc.pyの全テストがPASSしたら完了です。テストファイルは変更しないでください。"

# 避けたい例：範囲があいまいで、次工程まで踏み込む余地を与えてしまう
--message "calc.pyを改善してください。"
```

## まとめ

Claude Codeとの組み合わせは、「万能な1つのツールを選ぶ」のではなく、**タスクの性質に応じてツールを使い分ける**という考え方です。定型的で範囲の狭い作業はOrnith-1.0-9Bに任せてコストを抑えつつ、設計判断や最終確認はClaude Codeに残す。この役割分担が、無人運用の安全性とコスト効率を両立させる現実的な落としどころだと考えています。
